# 实验报告记录 Skill

## 目标

当用户要求“写实验报告”“记录实验结果”“整理实验进展”“把这次实验写到 docs 里”时，使用本 skill。

你的任务是：

1. 阅读用户提供的实验信息、终端输出、结果表格、代码变更说明或当前项目上下文。
2. 主动收集并详细记录本次实验配置，包括配置文件、启动命令、命令行参数、超参数、数据配置、评估配置、随机种子、环境、硬件、cache/checkpoint 路径、训练时间、测试时间、评估时间和各阶段耗时等。
3. 用中文写一份结构清晰、便于后续复习和复现实验的实验报告。
4. 自动在项目中寻找 `docs` 目录，并把报告保存到合适的位置。
5. 如果 `docs` 中已经存在实验报告目录，应优先放入已有目录。
6. 如果没有合适目录，可以创建统一目录，例如：
   - `docs/experiments/`
   - `docs/experiment_reports/`
   - `docs/实验报告/`
7. 绝对不要执行 `git commit`。
8. 绝对不要自动推送、创建 PR、修改远程仓库。

## 保存位置规则

执行前先检查项目中是否存在：

```bash
find . -maxdepth 3 -type d \( -name "docs" -o -name "doc" \)
```

然后在 `docs` 目录下寻找已有实验报告相关目录：

```bash
find docs -maxdepth 3 -type d | grep -Ei "experiment|report|实验|报告|results"
```

保存优先级：

1. 如果已有实验报告目录，放入已有目录。
2. 如果已有 `docs/experiments/`，放入其中。
3. 如果已有 `docs/experiment_reports/`，放入其中。
4. 如果已有 `docs/实验报告/`，放入其中。
5. 如果没有，则创建：

```bash
mkdir -p docs/experiments
```

不要把实验报告随意散落在 `docs/` 根目录，除非项目本身只有这种组织方式。

## 文件命名规则

文件名使用清晰、可排序的格式：

```text
YYYY-MM-DD_实验主题.md
```

例如：

```text
2026-05-13_two_tower_recall_smoke_test.md
2026-05-13_colbert_late_interaction_recall.md
2026-05-13_intersection_15hz_substep_collection.md
```

如果实验主题不明确，先根据上下文自动概括，不要反复追问。

如果同名报告已经存在，不要覆盖旧文件，应创建新文件，或在文件名后加序号：

```text
2026-05-13_two_tower_recall_smoke_test_v2.md
```

## 报告语言

报告必须使用中文。

技术术语可以保留英文，例如：

- Recall@500
- NDCG@20
- RRF
- Two-Tower
- ColBERT
- Smoke Test
- highway-env
- DQN
- PPO

## 报告结构

实验报告应尽量包含以下部分。

如果某些信息用户没有提供，可以写“本次未记录”或“当前信息不足”，不要编造。

第 4 节“实验设置”是每次实验报告的重点必填项。即使实验结果很简单，也要尽量把可复现实验所需的配置和耗时写清楚；不要只写“使用默认配置”或“见配置文件”，除非同时记录了配置文件路径和关键默认值。

```markdown
# 实验报告：<实验名称>

## 1. 实验目的

说明这次实验想验证什么问题。

重点回答：

- 为什么要做这个实验？
- 想确认模型、方法、数据或流程的哪一点？
- 这个实验在整个项目中的作用是什么？

## 2. 实验内容

说明本次具体做了什么。

可以包括：

- 使用了什么方法、模型或模块
- 修改了哪些代码或配置
- 执行了哪些训练、评估、采集或分析步骤
- 是否是 smoke test / medium / full 实验

## 3. 输入与数据

说明实验输入是什么，数据来自哪里。

需要尽量写清楚：

- 输入字段是什么
- 数据集是什么
- 数据规模是多少
- 训练集、验证集、测试集如何划分
- 每条样本代表什么
- 如果是推荐任务，说明 query/context/item/gold track 是什么
- 如果是轨迹预测任务，说明历史轨迹、未来轨迹、agent 数量、时间步等
- 如果是强化学习任务，说明环境、状态、动作、奖励等

## 4. 实验设置

详细记录关键参数、配置来源和运行设置。

必须尽量包括：

- 代码版本：分支名、commit、是否有未提交改动、关键改动文件。
- 运行入口：工作目录、完整运行命令、脚本路径、命令行参数、配置覆盖项。
- 配置文件：YAML / JSON / TOML / Python config 路径、实验名、配置 profile、重要字段摘录。
- 数据配置：数据集名称、输入文件/目录、train/valid/test 划分、采样比例、数据规模、过滤规则、预处理脚本或 cache。
- 模型配置：模型名称、backbone、checkpoint、embedding 维度、层数、hidden size、dropout、loss、negative sampling 等。
- 训练配置：epoch、batch size、learning rate、optimizer、scheduler、warmup、weight decay、gradient clipping、seed、precision、early stopping。
- 评估配置：指标、top-k、beam size、RRF 权重、rerank 设置、阈值、评估集范围、是否全量评估。
- 资源与环境：GPU 型号和数量、CUDA / PyTorch / Python / conda 环境、关键依赖版本、机器或远程服务器。
- 耗时记录：实验开始/结束时间、总耗时、数据预处理耗时、cache 构建耗时、训练耗时、每个 epoch 耗时、验证耗时、测试耗时、推理耗时、评估脚本耗时、保存 checkpoint 耗时、结果汇总耗时。
- 性能效率：样本吞吐量、step/s、it/s、每条样本平均耗时、GPU 利用率或显存峰值；如果日志中没有这些信息，明确写“本次未记录”。
- 产物路径：日志、tensorboard/wandb run、cache、checkpoint、预测文件、评估结果文件、失败案例文件。

如果用户提供了命令、配置文件片段或日志，尽量原样保留；如果能从项目文件或终端输出中可靠读到配置或耗时，应主动摘录关键字段。缺失的配置项和耗时项不要省略，写“本次未记录”或“当前信息不足”，并在后续计划中提示需要补齐。对于运行日志中的时间戳、progress bar、wandb/tensorboard 记录、`time` 命令输出、训练框架打印的 epoch time / eval time / test time，应尽量换算成易读格式并记录原始来源。

## 5. 实验结果

用表格整理主要结果。

例如：

| 指标 | 结果 |
|---|---:|
| Recall@20 | xx% |
| Recall@50 | xx% |
| Recall@100 | xx% |
| Recall@500 | xx% |

或者：

| Epoch | Train Loss | Val Loss |
|---:|---:|---:|
| 1 | x.xx | x.xx |
| 10 | x.xx | x.xx |

## 6. 对比与提升

如果用户提供了 baseline、上一次实验结果或旧方法结果，说明本次相较于之前提升在哪里。

重点回答：

- 相较于什么方法 / 哪次实验？
- 哪些指标提升了？
- 哪些指标没有明显变化？
- 提升是否稳定？
- 这个结果是否积极？
- 是否只是 smoke test，不能过度解读？

如果没有 baseline，则写：

> 当前缺少直接对比基线，因此本节暂不做定量提升判断。

## 7. 结果分析

用简洁语言解释结果意味着什么。

需要避免只堆数字，要说明：

- 结果是否符合预期
- 方法可能为什么有效
- 问题可能出在哪里
- 是否存在过拟合、样本过小、评估不充分等风险
- 对后续实验有什么启发

## 8. 当前结论

用 3 到 5 条总结本次实验结论。

示例：

1. 本次 smoke test 已验证代码链路可以正常运行。
2. 模型可以正常收敛，但验证集提升仍需进一步确认。
3. Recall@500 有一定改善，说明该召回通道具备继续扩展的价值。
4. 当前实验规模较小，不能直接代表 full devset 表现。

## 9. 后续计划

列出下一步建议。

例如：

- 扩展到 medium / full 数据规模
- 增加和 baseline 的统一对比
- 保存更完整的日志和 cache
- 做消融实验
- 调整超参数
- 检查失败案例
```

## 写作风格

要求：

- 中文
- 清晰
- 偏实验记录风格
- 方便后续复习
- 不要写成宣传稿
- 不要过度夸大结果
- 对 smoke test 结果要谨慎解释
- 如果信息不足，要明确写出来

推荐表达：

```text
本次实验的主要价值在于验证流程可行，而不是证明最终性能最优。
```

```text
由于当前只在较小规模数据上测试，结果只能作为方向性参考。
```

```text
该结果说明该方法具备继续扩大实验规模的价值。
```

## 禁止事项

绝对禁止：

```bash
git commit
git push
gh pr create
```

除非用户明确单独要求，否则不要：

- 自动提交代码
- 自动推送远程
- 自动创建 Pull Request
- 删除已有实验报告
- 覆盖旧报告
- 把实验报告随意放在 `docs/` 根目录
- 把英文报告当作最终输出

## 推荐执行流程

1. 先查看项目目录结构，确认是否有 `docs` 或 `doc`。
2. 查看 `docs` 下是否已有实验报告目录。
3. 根据用户提供的信息写报告。
4. 重点补齐“实验设置”：读取或摘录相关配置文件、运行命令、超参数、数据/环境/路径信息，以及训练、验证、测试、评估、预处理、cache 构建、推理等各阶段耗时；无法确认的项明确标注未记录。
5. 自动选择合适路径保存。
6. 检查文件是否写入成功。
7. 用 `git status --short` 检查工作区变化，但不要提交。
8. 向用户汇报保存路径，并说明没有执行 git commit。

## 完成后反馈

完成后告诉用户：

1. 报告保存路径。
2. 是否创建了新的实验报告目录。
3. 是否没有执行 git commit。

示例：

```text
已完成实验报告记录。

保存位置：
docs/experiments/2026-05-13_two_tower_recall_smoke_test.md

说明：
- 已自动放入 docs 下已有实验报告目录。
- 未执行 git commit。
```
